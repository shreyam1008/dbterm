Open-source handbook

dbterm is planned as a long-term public utility project. This page explains how contributors can participate effectively even if they are new to open source.

Project values

These principles should guide code decisions and product scope.

Terminal-first ergonomics

Every feature should preserve keyboard speed and low cognitive load.

Low-resource runtime

Keep binary and runtime overhead practical for laptops, servers, and constrained environments.

No lock-in stack

Avoid heavyweight runtime requirements for users who just need a robust DB client.

Contribution flow (beginner friendly)

Start small, stay precise, and link every change to a clear user outcome.

1. Set up locally

Fork the repository, clone your fork, and run the app in your terminal to understand current behavior.

2. Pick one scoped issue

Choose one bug fix or UX refinement, confirm expected behavior, and keep the first PR tight.

3. Ship with proof

Include reproducible steps, test notes, and screenshots or terminal output where relevant.

4. Iterate in public

Respond to review comments quickly and document final behavior in README/site docs when needed.

If you are new: documentation improvements, keyboard hint clarity, and repro-focused bug reports are high-value first contributions.

High-impact areas to work on

Connection UX

Validation messaging, better defaults, and cloud target setup guides.

Result handling

Large result ergonomics, sorting behavior, and export-oriented improvements.

Cross-platform polish

Windows terminal behavior, service command fallbacks, and shell-level edge cases.

Documentation

Onboarding docs, workflow recipes, troubleshooting playbooks, and examples.

Accessibility and readability

Color contrast, terminal theme clarity, and clear shortcut discoverability.

Release reliability

Improve automation checks, artifact verification, and installer robustness.

Release and maintenance model

Current automation keeps binaries and installers in sync with tagged releases.

  • Releases are automated from GitHub Actions on pushes to `main`.
  • Version metadata is read from `releases/versions.txt` in `<version>|<name>|<description>` format.
  • Assets and checksums are published to GitHub Releases for Linux, macOS, and Windows.
  • Install/update commands fetch the newest tagged release unless a specific version is requested.

Community channels

Use these paths so maintainers and contributors can collaborate efficiently.

  • Issues: bug reports and feature requests
  • Pull requests: implementation proposals
  • Releases: binary downloads and changelogs
  • Repository: source, scripts, and docs
